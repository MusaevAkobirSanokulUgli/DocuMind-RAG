🎯 Problem Statement

Why traditional document search fails for AI-powered Q&A

Organizations accumulate vast document libraries — PDFs, Word files, technical manuals, research papers — but retrieving precise answers from these documents remains difficult. Keyword search returns too many results without context; LLMs hallucinate without grounding; and most RAG solutions require expensive cloud API calls per query.

  • Keyword search returns document titles, not answers — users must read entire documents to find relevant information.
  • LLMs without retrieval "hallucinate" facts not in their training data, producing confidently wrong answers.
  • Cloud-based RAG solutions send sensitive document content to third-party APIs, creating data privacy risks.
  • No citation mechanism: even when AI gives a correct answer, users cannot verify which source document it came from.
  • High per-query cost at scale: cloud embedding APIs can cost thousands of dollars per month for large document sets.

✨ Key Features

Intelligent document Q&A with no external API dependencies

📂
Multi-format Document Upload
Supports PDF, DOCX, TXT, and Markdown. Text extraction is format-aware: PyMuPDF for PDFs, python-docx for Word documents.
✂
Recursive Chunking
Documents split into 512-character chunks with 50-character overlap. Overlap prevents context loss at chunk boundaries for better answer quality.
🔢
Local Embeddings
Uses all-MiniLM-L6-v2 (384 dimensions) running locally. Zero external API calls — your documents never leave your infrastructure.
🔍
HNSW Vector Search
ChromaDB's HNSW (Hierarchical Navigable Small World) index enables approximate nearest-neighbour search at 30–100ms latency without an LLM.
📎
Source Citations
Every answer includes citations linking back to the exact source document and chunk. Users can verify every claim against the original text.
📊
Analytics Dashboard
Recharts-powered analytics showing query volume, top documents, average relevance scores, and query latency trends over time.
No external API dependencies: The embedding model runs locally using sentence-transformers. Query latency is approximately 30–100ms without an LLM — adding an LLM generation step would add 1–5 seconds depending on hardware.

🚀 Quick Start

Get DocuMind running — the embedding model downloads automatically on first start

Step 1: Clone and enter the project

git clone https://github.com/your-org/rag-documind.git cd rag-documind

Step 2: Start all services (model downloads on first run ~90 MB)

docker compose up --build -d # Services starting: # ✓ frontend → http://localhost:3001 # ✓ dotnet-api → http://localhost:5001 # ✓ rag-engine → http://localhost:8000 # ✓ chromadb → http://localhost:8001 (internal)

Step 3: Upload a document via API

curl -X POST http://localhost:5001/api/documents/upload \ -F "file=@my_document.pdf" \ -F "title=My Document" # Returns: {"id": "abc123", "chunks": 42, "status": "indexed"}

Step 4: Ask a question

curl -X POST http://localhost:5001/api/query \ -H "Content-Type: application/json" \ -d '{"question": "What are the main findings?", "top_k": 5}' # Returns: {"answer": "...", "citations": [{"doc": "...", "chunk": "..."}]}

Step 5: Open the UI at http://localhost:3001

🔒 Security Considerations

Privacy-first design with local model execution

🏠 Local Model Execution

The all-MiniLM-L6-v2 model runs entirely on-premises. Document content and query text never leave your infrastructure.

📁 File Type Validation

Uploaded files are validated against an allow-list (PDF, DOCX, TXT, MD). File contents are inspected beyond extension — MIME type is checked.

📏 File Size Limits

Maximum upload size is enforced at the API gateway level (default 50 MB). Large files are rejected before reaching the parsing layer.

🗂 Vector Isolation

Each document's vectors are stored in a named ChromaDB collection. Deleting a document removes its vectors completely from the index.

🔗 Citation Integrity

Every answer includes chunk-level citations that map directly to source text, enabling users to verify AI-generated answers against originals.

🐳 Network Isolation

ChromaDB runs on an internal Docker network and is not exposed to the host. Only the frontend and .NET API are accessible externally.
