🎯 문제 정의

기존 문서 검색이 AI 기반 Q&A에 실패하는 이유

조직들은 방대한 문서 라이브러리를 축적합니다 — PDF, Word 파일, 기술 매뉴얼, 연구 논문 — 하지만 이러한 문서에서 정확한 답변을 검색하는 것은 여전히 어렵습니다. 키워드 검색은 컨텍스트 없이 너무 많은 결과를 반환하고, LLM은 근거 없이 환각을 일으키며, 대부분의 RAG 솔루션은 쿼리당 비싼 클라우드 API 호출을 필요로 합니다.

  • 키워드 검색은 답변이 아닌 문서 제목을 반환합니다 — 사용자가 관련 정보를 찾으려면 전체 문서를 읽어야 합니다.
  • 검색 없는 LLM은 훈련 데이터에 없는 사실을 "환각"하여 자신 있게 틀린 답변을 제공합니다.
  • 클라우드 기반 RAG 솔루션은 민감한 문서 내용을 서드파티 API에 전송하여 데이터 프라이버시 위험을 초래합니다.
  • 인용 메커니즘 부재: AI가 정확한 답변을 제공하더라도 사용자는 어느 소스 문서에서 왔는지 확인할 수 없습니다.
  • 대규모에서의 높은 쿼리 비용: 클라우드 임베딩 API는 대용량 문서 세트에서 월 수천 달러가 들 수 있습니다.

✨ 주요 기능

외부 API 의존성 없는 지능형 문서 Q&A

📂
다중 형식 문서 업로드
PDF, DOCX, TXT, Markdown을 지원합니다. 텍스트 추출은 형식에 맞게 처리됩니다: PDF에는 PyMuPDF, Word 문서에는 python-docx.
✂
재귀적 청킹
문서가 50자 겹침이 있는 512자 청크로 분할됩니다. 겹침은 청크 경계에서의 컨텍스트 손실을 방지합니다.
🔢
로컬 임베딩
로컬에서 실행되는 all-MiniLM-L6-v2(384차원)를 사용합니다. 외부 API 호출 제로 — 문서가 인프라를 벗어나지 않습니다.
🔍
HNSW 벡터 검색
ChromaDB의 HNSW 인덱스가 LLM 없이 30–100ms 지연시간으로 근사 최근접 이웃 검색을 가능하게 합니다.
📎
출처 인용
모든 답변은 정확한 소스 문서와 청크로 연결되는 인용을 포함합니다. 사용자는 모든 주장을 원본 텍스트와 대조 검증할 수 있습니다.
📊
분석 대시보드
쿼리 볼륨, 상위 문서, 평균 관련성 점수, 시간에 따른 쿼리 지연시간 추세를 보여주는 Recharts 기반 분석.
외부 API 의존성 없음: 임베딩 모델은 sentence-transformers를 사용하여 로컬에서 실행됩니다. 쿼리 지연시간은 LLM 없이 약 30–100ms입니다 — LLM 생성 단계를 추가하면 하드웨어에 따라 1–5초가 추가됩니다.

🚀 빠른 시작

DocuMind 실행하기 — 임베딩 모델은 첫 실행 시 자동으로 다운로드됩니다

1단계: 복제 및 프로젝트 진입

git clone https://github.com/your-org/rag-documind.git cd rag-documind

2단계: 모든 서비스 시작 (첫 실행 시 모델 ~90 MB 다운로드)

docker compose up --build -d # 서비스 시작 중: # ✓ frontend → http://localhost:3001 # ✓ dotnet-api → http://localhost:5001 # ✓ rag-engine → http://localhost:8000 # ✓ chromadb → http://localhost:8001 (내부)

3단계: API를 통해 문서 업로드

curl -X POST http://localhost:5001/api/documents/upload \ -F "file=@my_document.pdf" \ -F "title=내 문서" # 반환: {"id": "abc123", "chunks": 42, "status": "indexed"}

4단계: 질문하기

curl -X POST http://localhost:5001/api/query \ -H "Content-Type: application/json" \ -d '{"question": "주요 결과는 무엇인가요?", "top_k": 5}' # 반환: {"answer": "...", "citations": [{"doc": "...", "chunk": "..."}]}

5단계: http://localhost:3001에서 UI 열기

🔒 보안 고려사항

로컬 모델 실행을 통한 프라이버시 우선 설계

🏠 로컬 모델 실행

all-MiniLM-L6-v2 모델은 완전히 온프레미스에서 실행됩니다. 문서 내용과 쿼리 텍스트는 인프라를 벗어나지 않습니다.

📁 파일 유형 검증

업로드된 파일은 허용 목록(PDF, DOCX, TXT, MD)에 대해 검증됩니다. 파일 내용은 확장자 이상으로 검사됩니다 — MIME 유형도 확인됩니다.

📏 파일 크기 제한

최대 업로드 크기는 API 게이트웨이 수준에서 적용됩니다(기본값 50 MB). 큰 파일은 파싱 계층에 도달하기 전에 거부됩니다.

🗂 벡터 격리

각 문서의 벡터는 명명된 ChromaDB 컬렉션에 저장됩니다. 문서를 삭제하면 해당 벡터가 인덱스에서 완전히 제거됩니다.

🔗 인용 무결성

모든 답변은 사용자가 AI 생성 답변을 원본과 대조 검증할 수 있는 청크 수준 인용을 포함합니다.

🐳 네트워크 격리

ChromaDB는 내부 Docker 네트워크에서 실행되며 호스트에 노출되지 않습니다. 프론트엔드와 .NET API만 외부에서 접근 가능합니다.
