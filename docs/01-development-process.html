🤔 Problem & Solution

🚫 The Problem

Organizations waste 2–4 hours per day searching through documents. Traditional keyword search misses semantic meaning — searching "vehicle" won't find documents about "car" or "automobile".

✓ The Solution (RAG)

Retrieval-Augmented Generation: Document → Chunk → Embed → Store → Semantic Search → Answer with Citations. Understands meaning, not just keywords. Works fully offline.

🏗 3-Tier Architecture

Tier 1
React 18 Frontend — Port 3001
React 18.3 · TypeScript 5.7 · Vite 6.0 · Tailwind CSS 3.4 · React Router 6.28 · Recharts · Radix UI · Lucide React. Served via nginx in Docker.
Tier 2

Python 3.11 · FastAPI 0.115.6 · Uvicorn · sentence-transformers 3.3.1 · ChromaDB 0.5.23 · PyPDF2 3.0.1 · python-docx 1.1.2 · NumPy · PyTorch 2.5.1 · Pydantic 2.10.4. Core AI/ML pipeline.

🤖 AI/ML Pipeline — Step by Step

1
Text Extraction
PDF → PyPDF2 · DOCX → python-docx · TXT/MD → built-in with UTF-8 / Latin-1 / CP1252 fallback encoding. Max file size: 50MB. Supported formats: PDF, DOCX, TXT, MD.
2
Text Chunking — Recursive Character Splitting
chunk_size=512, chunk_overlap=50. 9-level separator hierarchy: paragraph → line → sentence → clause → word → character. Preserves semantic boundaries.
3
Embedding — all-MiniLM-L6-v2
22.7M parameters · 384 dimensions · ~80MB · runs locally on CPU · no API keys needed. batch_size=64 · L2-normalized vectors. Auto-downloaded from HuggingFace on first run.
4
Vector Storage — ChromaDB with HNSW Index
Cosine similarity space · persistent storage · batches of 500 · metadata stored alongside each chunk. HNSW (Hierarchical Navigable Small World) index for fast approximate nearest-neighbor search.
5
Retrieval & Reranking
Query → embed → HNSW search (over-retrieve top_k×2) → length-penalty reranking → deduplication → score threshold filter → return top_k results.
6
Answer Generation
Currently rule-based (no LLM required). Builds context from ranked chunks → formats answer with [Source N] citations. Optional: plug in OpenAI, Anthropic Claude, or Ollama for LLM-powered answers.

⚡ Performance Characteristics

Without LLM (Rule-based)

Query latency: 30–100ms
Pure vector search + context assembly. Production-ready for most use cases.

With LLM (Optional)

Query latency: 1–5 seconds
Adds natural language generation on top of retrieved context. Requires API key or local Ollama.

React app served by nginx. Maps port 3001. Depends on backend service.

backend

.NET 8 API. Maps port 5001. Connects to RAG engine via internal Docker network.

rag-engine

Python FastAPI. Maps port 8000. Shares uploaded-docs volume with backend.

Shared Volume The uploaded-docs Docker volume is mounted into both backend (for file storage) and rag-engine (for file processing). This allows the Python engine to read files uploaded through the .NET API without network transfer.

🤖 AI/ML Quvuri — Bosqichma-bosqich

1
Matn Ajratib Olish
PDF → PyPDF2 · DOCX → python-docx · TXT/MD → UTF-8 / Latin-1 / CP1252 zahira kodlash bilan o'rnatilgan. Maksimal fayl hajmi: 50MB. Qo'llab-quvvatlanadigan formatlar: PDF, DOCX, TXT, MD.
2
Matn Qismlarga Bo'lish — Rekursiv Belgi Ajratish
chunk_size=512, chunk_overlap=50. 9 darajali ajratuvchi ierarxiyasi: paragraf → satr → jumla → band → so'z → belgi. Semantik chegaralarni saqlaydi.
3
Vektorlashtirish — all-MiniLM-L6-v2
22.7M parametr · 384 o'lcham · ~80MB · CPU-da mahalliy ishlaydi · API kalit kerak emas. batch_size=64 · L2-normallashtirish. Birinchi ishga tushirishda HuggingFace-dan avtomatik yuklanadi.
4
Vektor Saqlash — HNSW Indeksi bilan ChromaDB
Kosinus o'xshashligi maydoni · doimiy saqlash · 500 ta partiyalar · har bir qism bilan birga metadata saqlanadi. HNSW (Hierarchical Navigable Small World) indeksi tez taxminiy qo'shni qidiruv uchun.
5
Qidirish va Qayta Tartiblash
So'rov → vektorlashtirish → HNSW qidiruv (ko'proq olish top_k×2) → uzunlik penalti bilan qayta tartiblash → dublikatlarni olib tashlash → ball filtri → top_k natijalarni qaytarish.
6
Javob Yaratish
Hozirda qoidaga asoslangan (LLM talab qilinmaydi). Tartiblangan qismlardan kontekst quriladi → [Manba N] iqtiboslari bilan javob formatlashadi. Ixtiyoriy: OpenAI, Anthropic Claude yoki Ollama ulash mumkin.

🤔 문제 및 해결책

🚫 문제점

조직들은 문서 검색에 하루 2~4시간을 낭비합니다. 기존 키워드 검색은 의미를 이해하지 못합니다 — "자동차"로 검색하면 "차량"이나 "승용차"에 관한 문서를 찾지 못합니다.

✓ 해결책 (RAG)

Retrieval-Augmented Generation: 문서 → 청크 → 임베딩 → 저장 → 의미 검색 → 출처 포함 답변. 단순 키워드가 아닌 의미를 이해합니다. 완전 오프라인 동작.

🤖 AI/ML 파이프라인 — 단계별 설명

1
텍스트 추출
PDF → PyPDF2 · DOCX → python-docx · TXT/MD → UTF-8 / Latin-1 / CP1252 폴백 인코딩으로 내장 처리. 최대 파일 크기: 50MB. 지원 형식: PDF, DOCX, TXT, MD.
2
텍스트 청킹 — 재귀적 문자 분할
chunk_size=512, chunk_overlap=50. 9단계 구분자 계층: 단락 → 줄 → 문장 → 절 → 단어 → 문자. 의미적 경계를 유지합니다.
3
임베딩 — all-MiniLM-L6-v2
2,270만 매개변수 · 384차원 · ~80MB · CPU에서 로컬 실행 · API 키 불필요. batch_size=64 · L2 정규화 벡터. 최초 실행 시 HuggingFace에서 자동 다운로드.
4
벡터 저장 — HNSW 인덱스를 사용한 ChromaDB
코사인 유사도 공간 · 영구 저장 · 500개 배치 · 각 청크와 함께 메타데이터 저장. HNSW(계층적 탐색 가능 소세계) 인덱스로 빠른 근사 최근접 이웃 검색.
5
검색 및 재순위화
쿼리 → 임베딩 → HNSW 검색 (초과 검색 top_k×2) → 길이 패널티 재순위화 → 중복 제거 → 점수 임계값 필터 → top_k 결과 반환.
6
답변 생성
현재 규칙 기반 (LLM 불필요). 순위화된 청크에서 컨텍스트를 구성 → [출처 N] 인용과 함께 답변을 형식화합니다. 선택적: OpenAI, Anthropic Claude 또는 Ollama 연결 가능.

🔧 개발 환경 요구 사항

핵심 기능에 외부 API 키가 필요하지 않습니다 임베딩 모델(all-MiniLM-L6-v2)은 로컬 CPU에서 실행됩니다. OpenAI, Anthropic 또는 유료 서비스가 필요하지 않습니다. 인터넷은 최초 실행 시 모델 다운로드(~80MB)에만 한 번 필요합니다.
